### C) Implement secure webhook endpoint

### D) Webhook processor with idempotency + retries

Implement a processor that:

1. Selects pending events:

* `status in ('received','failed')`
* `next_attempt_at IS NULL OR next_attempt_at <= now()`
* Order by `received_at ASC`
* Limit batch size (e.g. 25)

2. For each event:

* Increment `attempt_count`
* Acquire idempotency lock `stripe_idempotency_locks.key = "${stripe_event_id}:process"`

  * If exists, skip (already processed)
* Process based on event type:

  * `payment_intent.succeeded`
  * `payment_intent.payment_failed`
  * `payment_intent.canceled`
  * `charge.refunded` (or `charge.refund.updated` if used)
  * `account.updated` (optional but recommended for Connect readiness)
* Update `stripe_payment_state` for the payment_intent_id:

  * Upsert by `payment_intent_id`
  * Update status, charge_id, amount, currency, connected_account_id, metadata, last_event fields
* Also update your appÔÇÖs domain objects *without refactoring*:

  * If you have an ÔÇťinvoice paidÔÇŁ or ÔÇťdeposit paidÔÇŁ model, update it.
  * If you have job/invoice ids in Stripe metadata, map using that.
  * If mapping is missing, store as ÔÇťunlinkedÔÇŁ and expose in debug endpoint.

3. If processing succeeds:

* Mark webhook event status = `processed`, set `processed_at`

4. If processing fails with transient error:

* Mark status = `failed`
* Set `next_attempt_at = now() + backoff(attempt_count)`
* Store error text (truncated)

Backoff strategy:

* attempt 1: +30s
* attempt 2: +2m
* attempt 3: +10m
* attempt 4+: +1h (cap)

Define ÔÇťtransientÔÇŁ as:

* DB connection errors
* timeouts
* rate limit errors
* 5xx from Stripe on retrieve calls (if any)

Define ÔÇťnon-transientÔÇŁ as:

* missing required metadata mapping (still store, but mark processed with ÔÇťunlinkedÔÇŁ)
* invalid state transitions (log, mark processed)
