You are a senior full-stack engineer + QA lead working on GigAid.

Task: Implement a production-ready V1 churn prediction + retention trigger system, plus admin dashboard (“Copilot/Cockpit”) updates to monitor and act on churn risk.

Do NOT stub. Do NOT partially implement. Everything must be wired end-to-end: DB → APIs → cron/worker → UI → admin Copilot → logging.

========================
GOALS / SUCCESS CRITERIA
========================
1) Every user gets a nightly churn risk score computed from real product data.
2) Users are placed into tiers: Healthy, Drifting, At Risk, Critical.
3) Tier-based retention actions trigger automatically with idempotency and logging.
4) Admin Copilot dashboard shows:
   - Overall churn risk distribution
   - “At Risk” and “Critical” user lists
   - Each user’s signals, score breakdown, and action history
   - Buttons to trigger manual interventions (send nudge, grant trial, grant credit, mark contacted)
5) A JSON report endpoint exists for auditing the system health.
6) Must include tests for scoring and trigger logic.

========================
STEP 1 — DATABASE / MIGRATIONS
========================
Create the following tables (or Supabase equivalents). Add indexes for user_id and updated_at.

1) churn_metrics
- id (pk)
- user_id (fk)
- last_login_days int
- jobs_7d int
- msgs_7d int
- rev_30d numeric
- rev_delta numeric
- no_pay_14d boolean
- failed_payments int
- errors_7d int
- blocks_7d int
- limit_95_hits int
- downgrade_views int
- cancel_hover int
- score int
- tier text (Healthy|Drifting|AtRisk|Critical)
- computed_at timestamp
- created_at timestamp
- updated_at timestamp

2) retention_actions
- id (pk)
- user_id (fk)
- tier text
- action_type text (Nudge|Trial|Credit|FounderSave|LiteModeEnable|FollowUpWizard|Other)
- channel text (InApp|Email|SMS)
- payload jsonb (message template used, offer, etc.)
- status text (Queued|Sent|Failed|Skipped)
- idempotency_key text UNIQUE
- error text nullable
- sent_at timestamp nullable
- created_at timestamp

3) retention_playbooks
- id (pk)
- tier text
- priority int
- action_type text
- channel text
- template_key text
- delay_hours int
- enabled boolean
Seed defaults:
- Drifting: InApp Nudge → “Payday flow”
- AtRisk: InApp + Email → “7-day Pro trial”
- Critical: Email + InApp → “Founder save offer”

If there is already a notifications/messaging table, integrate instead of duplicating.

========================
STEP 2 — SIGNAL EXTRACTION (REAL DATA)
========================
Implement functions that compute the signals from existing tables:
- last_login_days: from user.lastLoginAt or auth/session table
- jobs_7d: count jobs created in last 7 days
- msgs_7d: count outbound messages in last 7 days
- rev_30d: sum payments received in last 30 days (Stripe + internal)
- rev_delta: rev_30d - prior_30d (previous 30 day window)
- no_pay_14d: true if no payments received in last 14 days
- failed_payments: count failed Stripe events or failed invoice payments in last 30 days
- errors_7d: count app errors/events if you have an events table; otherwise implement a lightweight server error logging table and start populating it
- blocks_7d: count paywall/limit blocks events
- limit_95_hits: count times user hit 95% warning in last 14 days
- downgrade_views: count visits to pricing/downgrade page (track in events)
- cancel_hover: count “cancel intent” events (open cancel modal, click cancel button but not confirm)

If an “events” or analytics log does not exist, create a minimal events table:
events(id, user_id, event_name, meta jsonb, created_at)
Instrument key events mentioned above.

========================
STEP 4 — NIGHTLY CRON / WORKER
========================
Create a scheduled job (cron) that runs nightly (e.g., 2am local).
For each active user:
1) compute signals
2) compute score + tier
3) upsert churn_metrics (computed_at)
4) trigger retention actions if tier >= Drifting

Must be idempotent:
- Do not spam the same user daily with the same action.
- Use idempotency_key = `${user_id}:${tier}:${action_type}:${YYYY-MM-DD}` or similar.
- Only send 1 action per user per day.

Include a backfill command:
`npm run churn:backfill --days=30`

========================
STEP 5 — RETENTION ACTIONS (PLAYBOOK EXECUTION)
========================
Implement a retention engine that:
- selects enabled playbooks for that tier ordered by priority
- evaluates eligibility (e.g., user hasn’t received this action in last X days)
- enqueues the action (retention_actions row status=Queued)
- sends via appropriate channel (InApp/Email/SMS) using existing messaging infrastructure
- updates retention_actions to Sent/Failed

Default actions:
Drifting:
- InApp Nudge: “Want more bookings this week? Start the Get Paid Today flow.”
- CTA deep link to /onboarding/payday

AtRisk:
- InApp + Email: “Let’s get you booked this week”
- Offer: 7-day Pro trial (implement entitlement override with expiry)
- CTA to Lite Mode and Templates

Critical:
- InApp + Email: “We’ll help you get set up”
- Offer: 1 free month credit OR 14-day Pro trial
- Flag user for “Founder Save” list in admin

Implement entitlement overrides table if needed:
plan_overrides(user_id, override_type, expires_at, created_at)
